## Why this exists

The repository carries two parallel planning tracks that have drifted out of
sync with the code:

- **Compliance / global-alignment** (`docs/superpowers/plans/`): ~90% complete.
  A handful of concrete items remain to close the
  `chore/compliance-alignment-2026-06-19` branch.
- **Product** (`docs/project/PROJECT_PLAN.md`, `MVP.md`, `docs/adr/`): the
  documentation describes a 13-phase, 30-week enterprise build with 13 Accepted
  ADRs, but only ~15-20% is implemented. What exists is the storage schema layer
  plus a one-directional PP XML exporter. Extraction, classification, XML
  import, round-trip, Alembic migrations, and the CLI entry point do not exist.

This roadmap reconciles the two. It closes compliance first, makes the ADRs and
roadmap honest, then builds the first end-to-end product slice (XML round-trip).

## Sequencing summary

```text
Phase A (compliance close)  ->  merge branch
        |
Phase B (ADR + roadmap honesty)
        |
Phase C (XML round-trip walking skeleton)  <-- first real product slice
        |
Phase D (first broker ingestion)  <-- MVP feature delivery starts
        |  (taxonomy foundation, PR #92, is shared by C and D)
        |
Phase E (IBOR/ABOR navigation + multi-entity)  <-- ADR-016; after #92 merges
```

Phases A and B are small and can be done in the current branch context or a
short-lived `chore/` and `docs/` branch respectively. Phase C is the first
phase that warrants a dedicated feature branch and an ADR (ADR-014). Phase D
and beyond reopen the deferred ADRs one feature at a time, re-statusing each
back to `Accepted` only when its code lands. Phase E (ADR-016) is the cross-book
IBOR/ABOR layer; it is captured now but built after the PR #92 taxonomy
foundation lands on `main`.
